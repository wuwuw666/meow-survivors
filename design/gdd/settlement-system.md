# 结算系统 (Settlement System)

> **Status**: In Design
> **Author**: [user + agents]
> **Last Updated**: 2026-04-15
> **Implements Pillar**: 成长的爽感 + 可爱即正义

---

## Overview

结算系统负责在单局结束时展示结果、汇总统计，并把这局表现转化为**塔局外成长资源**。

在当前 `C-Tangy` 成长结构中，结算系统不只是“失败后弹个面板”，而是以下三件事的桥梁：

1. 让玩家理解这局发生了什么
2. 让玩家获得清晰的再来一局动力
3. 让本局表现转化为塔的长期成长进度

因此，结算系统当前的定位是：

- 展示单局结果与核心统计
- 发放塔成长资源
- 提供重开或返回菜单入口

结算系统**不负责**：

- 角色局内升级
- 塔局内改造
- 直接执行塔局外成长升级逻辑

---

## Design Goal

结算系统要让玩家在结束一局时产生这样的感觉：

- “这局虽然输了，但我还是推进了塔体系”
- “我知道自己是怎么输的，也知道下一局想怎么调整”
- “这局打到更后面是有长期价值的”

也就是说，结算必须同时承担：

- 情绪收尾
- 结果解释
- 长期成长反馈

---

## Core Loop

```text
玩家死亡 / 通关条件达成
-> 战斗结束
-> 收集本局统计
-> 计算结算奖励
-> 展示结算面板
-> 玩家查看资源与表现
-> 选择进入塔成长界面 / 再来一局 / 返回菜单
```

---

## Settlement Outcome

### 结算触发条件

MVP 结算触发条件：

- 玩家生命归零
- 达成当前短局终点（例如 Wave 10 完成）

### 结算结果类型

| 结果 | 说明 |
|---|---|
| `Defeat` | 玩家在波次中被击败 |
| `Clear` | 玩家完成当前短局目标 |

Clear 与 Defeat 的差异应主要体现在：

- 标题文案
- 奖励倍率
- 鼓励重开的情绪表达

---

## Tracked Stats

结算系统记录以下 MVP 统计：

| 字段 | 说明 |
|---|---|
| `waves_cleared` | 完成波次数 |
| `enemies_killed` | 总击杀数 |
| `hero_level` | 本局角色最高等级 |
| `towers_placed` | 本局建造塔数量 |
| `elite_mods_taken` | 本局获得的塔改造次数 |
| `coins_earned` | 本局获得金币 |
| `run_duration_sec` | 本局时长 |
| `result_type` | Defeat / Clear |

MVP 不强求一开始就追踪过多花哨统计，优先保证：

- 波次进度
- 角色成长
- 塔布置
- 塔改造
- 资源收益

## Reward Model

### 规则 1：结算发放塔成长资源

结算奖励的核心输出是：

- `tower_scrap` 或类似塔成长资源

它用于后续局外系统：

- 解锁新塔
- 升级塔基础属性
- 解锁塔分支或塔成长节点

### 规则 2：奖励按本局表现计算

MVP 奖励建议由以下部分组成：

```text
settlement_reward =
    base_reward
    + wave_reward
    + clear_bonus_if_applicable
    + optional_performance_bonus
```

建议拆分：

- `base_reward`：无论输赢都给的基础资源
- `wave_reward`：按到达波次增加
- `clear_bonus`：通关额外奖励
- `performance_bonus`：可选，按精英改造次数或特殊表现给少量奖励

### 规则 3：短局也要有长期推进感

即使玩家较早失败，也应获得少量塔成长资源。

设计意图：

- 避免“白打了”的感觉
- 支撑塔体系的长期成长
- 让玩家愿意持续试不同塔阵

---

## Example Reward Formula

MVP 建议公式：

```text
tower_scrap =
    5
    + waves_cleared * 2
    + clear_bonus
```

其中：

- Defeat：`clear_bonus = 0`
- Clear：`clear_bonus = 10`

示例：

| 结果 | 波次 | 奖励 |
|---|---:|---:|
| Defeat | 3 | 11 |
| Defeat | 7 | 19 |
| Clear | 10 | 35 |

这版公式的目标是：

- 前中期失败也有稳定推进
- 通关有明显额外奖励
- 奖励曲线不会过陡，避免局外成长压过局内决策

## UI Structure

MVP 结算面板建议展示以下区域：

### 1. 结果标题

示例：

- `猫咪先撤一步...`
- `差一点就守住了！`
- `成功守住猫咪据点！`

### 2. 本局摘要

展示：

- 到达波次
- 本局时长
- 角色等级
- 塔建造数
- 塔改造数

### 3. 结算奖励

重点展示：

- 本局获得的 `tower_scrap`
- 奖励来源拆分

示例：

```text
基础奖励: 5
波次奖励: 14
通关奖励: 10
总计: 29 Tower Scrap
```

### 4. 操作按钮

- `前往塔成长`
- `再来一局`
- `返回菜单`

`前往塔成长` 的设计目标：

- 把本局获得的 `tower_scrap` 立刻接到长期成长
- 让玩家形成“打一局 -> 看奖励 -> 推进塔体系”的闭环
- 不强制每次都进入，但要作为明确入口存在

MVP 回流路径建议：

```text
结算面板
-> 前往塔成长
-> 购买或查看局外成长节点
-> 返回主菜单或开始下一局
```

---

## Messaging Rules

结算文案要做到：

- 输了也不打击人
- 赢了要有清晰高潮
- 明确告诉玩家“这局资源会喂给塔成长”

玩家应该在结算时明确理解：

- 角色成长是本局内的
- 塔改造是本局内的
- 结算奖励是局外塔成长的输入
- 若玩家选择进入塔成长界面，应能直观看到本局奖励如何转成长期进度

## Dependencies

### 上游依赖

- 生命值系统：提供失败触发
- 波次系统：提供波次进度
- XP 系统：提供角色等级
- 塔系统：提供放塔统计
- 塔改造系统：提供改造获取统计
- 金币系统：提供资源统计

### 下游依赖

- 局外塔成长系统：消费 `tower_scrap`
- UI 系统：展示结算面板
- UI 系统：展示结算面板与前往塔成长入口
- 存档系统（未来）：保存局外资源

## Tuning Knobs

| 参数 | 默认值 | 说明 |
|---|---:|---|
| `base_reward` | 5 | 基础结算奖励 |
| `reward_per_wave` | 2 | 每波奖励 |
| `clear_bonus` | 10 | 通关额外奖励 |
| `death_delay_sec` | 1.0 | 死亡到结算的延迟 |

---

## Edge Cases

| 编号 | 情况 | 处理方式 |
|---|---|---|
| EC-01 | 很早失败 | 仍然发放基础奖励 |
| EC-02 | 结算时同时触发其他面板 | 结算面板优先 |
| EC-03 | 通关与升级同时触发 | 先完成升级，再进入结算 |
| EC-04 | 奖励结算后立即重开 | 保证奖励已记录，再进入新局 |

## Acceptance Criteria

| ID | 验证项 | Pass 标准 |
|---|---|---|
| AC-ST-01 | 死亡触发结算 | 玩家死亡后正确进入结算面板 |
| AC-ST-02 | 通关触发结算 | 完成短局目标后正确进入结算面板 |
| AC-ST-03 | 统计展示 | 能正确显示波次、等级、建塔与改造统计 |
| AC-ST-04 | 资源发放 | 能正确结算并展示 `tower_scrap` |
| AC-ST-05 | 长期成长入口清晰 | 玩家能理解奖励用于塔局外成长 |
| AC-ST-06 | 操作出口清晰 | 结算后可重开或返回菜单 |
