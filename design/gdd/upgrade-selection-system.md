# 升级选择系统 (Upgrade Selection System)

> **Status**: In Design
> **Author**: [user + agents]
> **Last Updated**: 2026-04-10
> **Implements Pillar**: 成长的爽感 + 策略有深度

---

## Overview

升级选择系统负责在角色升级时弹出候选项，并让玩家从中选择一项角色局内强化。

在当前 `C-Tangy` 成长结构中，这个系统只服务于**角色局内成长**。它不再承担塔成长入口，不再提供塔数值升级，也不提供塔改造件。塔的局内花样由 `tower-mod-system` 负责，塔的长期成长由结算与局外成长系统负责。

系统职责：

- 监听 XP 系统的升级请求
- 暂停战斗并弹出升级面板
- 从角色升级池请求候选项
- 接收玩家选择并应用角色强化
- 关闭面板并恢复游戏

---

## Player Fantasy

玩家应该感受到：

- “我升级了，这次我要把角色往哪种支援方向养”
- “我不是在乱点数值，而是在决定这局角色怎么帮助防线”
- “选完以后，角色的移动、补漏、debuff 或生存立刻更顺”

玩家不应该感受到：

- “XP 升级为什么又在强化塔”
- “角色升级和塔改造混成一团”
- “三选一里大部分都是与当前角色玩法无关的垃圾项”

---

## Design Goal

升级选择系统的首要目标是塑造**角色在本局中的支援身份**。

默认优先级：

1. `Hero Support`
2. `Hero Mobility`
3. `Hero Survival`
4. `Hero Direct Combat`

角色升级允许少量直接战斗强化，但不能长期压过塔阵在防线中的主角地位。

---

## Core Loop

```text
XP 系统触发 level_up_requested
-> 游戏暂停
-> 升级池返回 3 个角色升级候选
-> 玩家选择 1 项
-> 应用到角色相关系统
-> 确认升级完成
-> 恢复游戏
```

---

## Detailed Rules

### 规则 1：候选只来自角色升级池

升级选择系统只展示角色升级项，不展示：

- 塔伤害升级
- 塔攻速升级
- 塔范围升级
- 塔改造件
- 局外成长节点

### 规则 2：升级面板暂停战斗

MVP 默认在升级面板打开期间暂停战斗流程，包括：

- 敌人移动
- 攻击冷却
- 波次计时
- 拾取物移动

设计意图：

- 避免玩家在升级时被迫承受战斗压力
- 保持每次升级都是有分量的决策瞬间

### 规则 3：默认三选一

MVP 默认为：

- 每次角色升级提供 `3` 个候选
- 玩家必须选择 `1` 个
- 不提供跳过
- 不提供 reroll

### 规则 4：升级分类显示

每个候选项必须明确标注其分类，至少覆盖以下类别：

- `Hero Support`
- `Hero Mobility`
- `Hero Survival`
- `Hero Direct Combat`

显示目标是让玩家一眼看懂：

- 强化的是哪种角色职责
- 这项升级主要帮助什么战斗行为

### 规则 5：升级应用范围

角色升级只应用到角色相关系统，例如：

- 自动攻击
- 移动系统
- 生存相关组件
- debuff / 支援逻辑

不直接应用到：

- 塔系统基础属性
- 塔改造系统
- 结算资源

## Upgrade Categories

### Hero Support

强化角色对防线的支援价值。

示例：

- 对被减速敌人额外造成伤害
- 英雄命中目标时施加易伤
- 在辅助塔覆盖区内获得额外支援能力

### Hero Mobility

强化角色移动与补位能力。

示例：

- 移速提升
- 冲刺冷却缩短
- 靠近漏怪时获得短时机动加成

### Hero Survival

强化角色容错与持续站场能力。

示例：

- 最大生命提高
- 受击减伤
- 紧急恢复

### Hero Direct Combat

强化角色的直接输出能力。

示例：

- 角色伤害提升
- 角色攻速提升
- 弹体数量增加

约束：

- 该类应为辅，不应成为默认最优解

## Core Interface

- `open_upgrade_panel(level: int) -> void`
- `is_panel_active() -> bool`
- `apply_selected_upgrade(upgrade_id: String) -> void`
- `signal upgrade_panel_opened(level: int)`
- `signal upgrade_chosen(upgrade_id: String)`
- `signal upgrade_panel_closed()`

---

## Dependencies

### 上游依赖

- XP 系统：发出 `level_up_requested`
- 升级池系统：返回角色升级候选

### 下游依赖

- 自动攻击系统：消费输出类升级
- 移动系统：消费机动类升级
- 生存系统：消费生存类升级
- UI 系统：渲染升级面板与分类标签

---

## Edge Cases

| 编号 | 情况 | 处理方式 |
|---|---|---|
| EC-01 | 面板打开时再次触发升级 | 当前面板优先，剩余 XP 待后续处理 |
| EC-02 | 候选不足 3 个 | 展示所有合法候选，不显示空白伪选项 |
| EC-03 | 玩家死亡与升级同时发生 | Game Over 优先 |
| EC-04 | 波次提示与升级冲突 | 升级面板优先，波次提示延后 |

## Acceptance Criteria

| ID | 验证项 | Pass 标准 |
|---|---|---|
| AC-US-01 | 升级触发 | 角色升级时正确弹出面板 |
| AC-US-02 | 候选来源正确 | 候选只来自角色升级池 |
| AC-US-03 | 战斗暂停 | 面板期间战斗暂停 |
| AC-US-04 | 分类清晰 | 玩家能看懂升级是支援、机动、生存还是直战 |
| AC-US-05 | 角色专属应用 | 选择升级后只强化角色，不直接强化塔 |
