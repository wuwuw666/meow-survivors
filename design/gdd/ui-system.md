# UI 系统 (UI System)

> **Status**: In Design  
> **Author**: [user + agents]  
> **Last Updated**: 2026-04-09  
> **Implements Pillar**: 可爱即正义（UI 风格）+ 成长的爽感（升级反馈）+ 策略有深度（塔阵信息清晰）

## Overview

UI 系统（UI System）负责向玩家展示《喵族幸存者》的所有关键局内信息，并承载升级、塔位、波次和结算等核心交互。

在当前设计方向中，UI 不再只服务“角色战斗状态”，而是明确服务于以下主轴：

- **塔阵构筑是否清晰**
- **经验升级是否顺手**
- **波次节奏是否可感知**
- **玩家是否知道自己这一局的防线成型到了什么程度**

当前版本里，UI 的优先级应围绕“塔防主导、角色支援、短局节奏、经验成长”来设计，而不是围绕传统幸存者 HUD 去堆信息。

**核心职责**：

- 展示核心 HUD 信息
- 提供升级选择界面
- 提供塔位状态与塔信息反馈
- 提供伤害、金币、经验等短时反馈
- 提供波次提示与 Boss 提示
- 提供结算与重开入口

**核心接口**：

- `update_hp(current: int, max: int) -> void`
- `update_coins(amount: int) -> void`
- `update_xp(current: int, needed: int) -> void`
- `update_wave(current: int, total: int) -> void`
- `update_tower_slots(slot_states: Array) -> void`
- `show_upgrade_panel(options: Array[Dictionary]) -> void`
- `show_tower_info(tower_data: Dictionary) -> void`
- `damage_number(position: Vector2, value: int, is_crit: bool, color: String) -> void`

## Player Fantasy

**情感目标：一眼看懂局势，且每次成长都被好好呈现。**

玩家应该感受到：

- “我一眼就知道这局现在稳不稳。”
- “我知道自己距离下一次升级还有多远。”
- “我知道哪几个塔位已经成型了。”
- “这一波结束了，我能立刻明白接下来要准备什么。”
- “升级弹出来时，我能迅速理解这是在强化角色、强化塔，还是强化联动。”

玩家不应该感受到：

- “我现在到底是在靠角色还是靠塔在撑？”
- “我有几个关键塔位已经用了？”
- “为什么升级了，但我没感觉？”
- “波次什么时候结束、Boss 什么时候来，我都看不清楚。”
- “屏幕上信息很多，但没有哪个是真正帮助决策的。”

## Design Direction

### UI 要优先服务塔阵信息

当前版本不是纯幸存者界面，而是塔防主导界面。  
因此 UI 必须先帮玩家回答这些问题：

- 哪些塔位是空的
- 哪些塔位已经成型
- 这波压力来自哪里
- 下一次升级会如何帮助防线

### 经验升级是主成长入口

升级面板的触发与强调，应围绕经验系统展开。  
波次 UI 可以提供短暂整理感，但不应该把“波次结算”做成主要升级来源的错觉。

### 波次提示要短、清楚、强节奏

波次提示不应该占屏太久，但必须让玩家清楚知道：

- 新的一波开始了
- 这一波已经清掉
- Boss 波即将到来

## Detailed Rules

### 规则 1：UI 层级架构

```text
RootUI
├── HUD_Layer
│   ├── HPBar
│   ├── XPBar
│   ├── CoinLabel
│   ├── WaveLabel
│   └── TowerSlotIndicator
├── PANEL_Layer
│   ├── UpgradePanel
│   └── TowerInfoPanel
├── FEEDBACK_Layer
│   └── FloatingFeedbackContainer
└── OVERLAY_Layer
    ├── BossWarningOverlay
    └── SettlementOverlay
```

MVP 中的 UI 重点应是：

- **HUD 层**
- **升级面板**
- **塔位状态显示**
- **短时反馈**
- **Boss / 结算覆盖层**

### 规则 2：HUD 信息优先级

HUD 不应平均对待所有信息，而应按优先级排列：

1. **波次状态**
2. **经验升级进度**
3. **塔位状态**
4. **生命值**
5. **金币**

原因：

- 当前项目的主要决策不只是“活着”，而是“防线是否成型”
- 所以塔位状态和波次状态的重要性高于传统幸存者 HUD

### 规则 3：血条（HP Bar）

血条仍然重要，但它的设计目标是让玩家快速判断“我还能不能补位”，而不是成为唯一视觉中心。

建议：

- 位置：左上角
- 尺寸：中等
- 颜色分段：
  - `> 50%` 正常色
  - `25% - 50%` 警告色
  - `< 25%` 红色脉冲
- 更新：使用平滑过渡

### 规则 4：经验条（XP Bar）

经验条是当前版本最关键的成长 UI 之一。

建议：

- 位置：左上区域，紧邻 HP 区域
- 必须清楚展示：
  - 当前经验
  - 升级需求
  - 当前等级
- 升级瞬间要有明显反馈：
  - 条满
  - 闪烁
  - 弹出升级面板

设计要求：

- 经验条必须让玩家意识到“成长主要来自经验”
- UI 不能把这个成长入口做得比波次提示还弱

### 规则 5：金币显示

金币的作用是支撑布防节奏，因此金币 UI 应服务“我现在能不能补塔”。

建议：

- 位置：左下或左侧下方
- 数值变化时要有轻量跳动
- 花钱时要有明确负反馈
- 不需要过度花哨，但要够清楚

### 规则 6：波次信息（Wave Label）

波次 UI 应该承担节奏提示责任。

显示状态建议：

- **进行中**：`第 X 波`
- **清场完成 / 整理中**：`第 X 波完成`
- **Boss 前**：`第 10 波 - BOSS 来袭`

视觉要求：

- 位置：顶部中央
- 进入新波时短暂强调
- Boss 波前必须明显强化
- 不应长时间挡住视野

### 规则 7：塔位指示（Tower Slot Indicator）

这是当前版本非常关键的 UI 组件。  
它的目标不是装饰，而是帮助玩家理解“我的防线结构”。

建议展示：

- 空塔位
- 已占用塔位
- 输出塔 / 辅助塔类型
- 是否为关键增益位或主火力位（后续可扩展）

玩家应能通过这一块迅速判断：

- 哪个塔位还没成型
- 哪个塔位已经放了关键塔
- 当前防线偏输出还是偏辅助

MVP 建议：

- 顶部右侧或右上安全区
- 使用简化图标而不是复杂面板

### 规则 8：塔信息面板（Tower Info Panel）

当前项目既然强调塔是主角，UI 就必须允许玩家查看塔信息。

MVP 面板可精简为：

- 塔名称
- 塔类型（输出 / 辅助）
- 当前作用（伤害 / 控制效果 / 增益）
- 关键数值

目标不是做复杂经营面板，而是让玩家能回答：

- 这座塔现在在干什么
- 它值不值得继续围绕它构建防线

### 规则 9：升级选择面板

升级面板是最重要的交互面板之一。

当前设计要求：

- 由经验升级触发
- 居中弹出
- 3-4 选 1
- 暂停游戏

最关键的是文案与分类要清楚。  
升级最好能被玩家快速识别为：

- **角色支援型**
- **输出塔强化型**
- **辅助塔 / 联动强化型**

这样玩家才能围绕防线做决策，而不是只看数值涨幅。

### 规则 10：短时反馈（Damage / Coins / XP）

反馈层用于强化成长与命中感，但不能抢走 HUD 可读性。

需要支持：

- 普通伤害数字
- 暴击伤害数字
- 金币获取反馈
- 经验获取反馈

当前方向建议：

- 伤害：敌人头顶出现
- 金币：更靠近拾取点与 HUD 金币标签方向
- 经验：更靠近角色与 XP 条心智关联

### 规则 11：Boss 提示覆盖层

Boss 波是整局高潮，必须有单独 UI 处理。

建议：

- 全屏弱遮罩
- 中央红色 / 暖色 Boss 提示
- 动画时长短，但存在感强

目标：

- 明确告诉玩家“这是最后高潮，不是普通波”

### 规则 12：结算画面

短局游戏的结算画面要承担“让我再开一局”的责任。

MVP 结算界面至少应展示：

- 存活 / 失败波次
- 玩家等级
- 金币或总收益
- 本局关键塔阵表现摘要（后续可扩展）
- 重开按钮

设计目标：

- 不做冗长结算
- 快速提供复盘与重开的动机

## Visual Direction

### 风格要求

- 温暖、圆润、可爱
- 不阴暗、不血腥
- 不做过度工业化或科技风

### 视觉层级

- 波次 / Boss 提示：最强强调
- 升级面板：第二强调
- 塔位状态：持续可见但不过分抢眼
- HP / XP / 金币：清晰稳定
- 飘字反馈：短时出现，不覆盖关键 UI

### 字体与图标

- MVP 可用 Godot 默认字体
- 图标尽量圆润、简化、可快速辨识
- 输出塔 / 辅助塔应在视觉上可立即区分

## Window Scaling

为保证 UI 在不同窗口尺寸下仍可用，遵循：

1. 参考分辨率：`1280 x 720`
2. HUD 使用明确锚点
3. 居中组件使用中心锚点
4. 面板类 UI 使用 `VBoxContainer` / `HBoxContainer`
5. 所有弹出面板必须设置 `min_size`

额外要求：

- 21:9 下保持 HUD 不被拉得过散
- 小窗口下优先保证升级面板、波次信息、塔位信息仍可读

## Formulas

### 公式 1：金币反馈动画

```text
start = world_pickup_position
end = HUD_coin_label_position
duration = 0.8s - 1.0s
```

### 公式 2：经验升级反馈

```text
xp_bar_fill -> flash -> panel_open
```

设计意义：

- 让玩家把“条满”和“升级选择”在脑中自然连起来

### 公式 3：波次提示显示时长

```text
normal_wave_banner = 0.8s - 1.2s
boss_banner = 1.5s - 2.0s
```

### 公式 4：飘字数量控制

```text
same_region_feedback_cap = 3
merge_window = 0.1s
```

用于防止反馈淹没画面。

## Edge Cases

| 编号 | 边界情况 | 处理方式 |
|------|----------|----------|
| EC-01 | 升级面板打开时玩家死亡 | 关闭升级面板，进入结算 |
| EC-02 | 多次升级连续触发 | 排队展示，不同时弹多个面板 |
| EC-03 | 波次提示与升级面板冲突 | 升级面板优先，波次提示缩短或延后 |
| EC-04 | Boss 提示与塔信息面板冲突 | Boss 提示优先级更高 |
| EC-05 | 飘字过多遮挡塔位信息 | 合并 / 限流反馈 |
| EC-06 | 小窗口导致塔位指示过挤 | 简化图标与缩短文本 |
| EC-07 | 玩家金币不足却频繁尝试放塔 | 给出明确但轻量的提示，不刷屏 |

## Dependencies

### 上游依赖

| 系统 | 依赖类型 | 接口 | 说明 |
|------|----------|------|------|
| **生命值系统** | 硬依赖 | `hp_changed` | 更新血条 |
| **经验系统** | 硬依赖 | `xp_changed`, `level_up` | 更新经验条与打开升级面板 |
| **金币系统** | 硬依赖 | `coins_changed` | 更新金币显示 |
| **波次系统** | 硬依赖 | `wave_started`, `wave_cleared`, `boss_wave_started` | 更新节奏提示 |
| **防御塔系统** | 软依赖 | `tower_placed`, `tower_removed`, `tower_projectile_hit` | 更新塔位状态与塔信息 |

### 下游依赖

| 系统 | 依赖类型 | 接口 | 说明 |
|------|----------|------|------|
| **升级系统** | 软依赖 | 玩家选择升级选项 | UI 回传选择 |
| **塔位放置系统** | 软依赖 | 玩家点击或查看塔位 | UI 协助表达状态 |
| **结算系统** | 软依赖 | 展示结果 | UI 负责最终呈现 |

## Tuning Knobs

| 参数名 | 默认值 | 安全范围 | 说明 |
|--------|--------|----------|------|
| `wave_banner_duration` | 1.0s | 0.6-1.5s | 普通波提示时长 |
