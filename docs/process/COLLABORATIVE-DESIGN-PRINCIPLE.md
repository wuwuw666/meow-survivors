# Collaborative Design Principle

## Purpose

本项目采用“用户驱动协作”模式。
无论是设计、实现、重构还是文档整理，代理和协作者都应先理解、再提案、再执行。

## Core Rule

默认流程不是“用户一句话，代理直接写完整结果”。
默认流程是：

1. 理解现状
2. 提出选项或草稿
3. 用户确认方向
4. 执行写入或修改
5. 说明结果与影响

## For Design Work

涉及玩法、数值、系统定义时：

- 先读对应 `design/gdd/*.md`
- 优先解释当前设计，而不是直接重写
- 如果文档不足，先补设计草稿
- 设计确认后，再进入实现

设计工作中应优先回答这些问题：

- 这个系统的玩家体验目标是什么
- 它依赖哪些已有系统
- 它会影响哪些现有实现
- 它是新增设计，还是修正文档与代码的不一致

## For Implementation Work

涉及代码修改时：

- 先说明将修改哪些文件
- 说明这些改动对应哪个系统文档
- 如果是多文件改动，先给出变更摘要
- 如果实现偏离现有 GDD，必须先指出

## Design vs Implementation Conflict

当文档与代码不一致时，不默认任何一方绝对正确。

先做三件事：

1. 指出不一致点
2. 说明影响
3. 让用户决定优先修文档还是修实现

## Incremental Drafting

长文档或大功能应采用分阶段协作：

- 先给目录或结构
- 再给关键段落或关键模块草稿
- 用户确认后再继续扩展
- 最终确认后再写入文件

这样可以减少返工，也方便保持项目方向一致。

## Approval Rule

以下情况默认需要先展示草稿或变更摘要：

- 新建设计文档
- 重写项目规范
- 多文件代码改动
- 影响系统边界的重构
- 改动核心玩法或关键数值

## Practical Standard For This Project

结合 Meow Survivors 当前状态，协作时额外遵守：

- `design/gdd/` 是玩法设计主来源
- `main_game.gd` 是当前总控，但不是继续堆逻辑的默认落点
- 设计文档整理与目录重组优先“小步移动、明确映射”，避免一次性大搬家
- 原型代码允许存在，但需要标清是否为临时方案

## Good Collaboration Example

好的协作应该像这样：

- 先说“我看了哪些文档和哪些代码”
- 再说“我建议怎么改，以及为什么”
- 再说“会改哪些文件”
- 最后等确认后再写

## Bad Collaboration Example

不好的协作包括：

- 没看 GDD 就直接改玩法
- 没说明影响范围就直接多文件修改
- 发现文档与代码冲突却自行决定
- 把临时原型逻辑当成正式架构继续扩张

## Goal

这个原则的目标不是拖慢效率，而是避免：

- 设计漂移
- 代码与文档脱节
- 重构失控
- 项目结构越来越混乱

对这个项目来说，协作的核心价值是：
在持续推进原型的同时，让设计、实现和文档逐步对齐。
